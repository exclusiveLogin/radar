# ADR-001: CPU LLM enricher delivery (Ollama + Docker)

- Status: Proposed
- Date: 2026-05-08
- Owners: Radar team

## 🔎 Контекст

Сейчас гео-обогащение в pipeline работает как `cache -> dadata -> nominatim -> llm`, где `llm` пока заглушка.

Цель: добавить легкий LLM-компонент для разбора "сложных" текстовых сообщений на CPU, с простым развертыванием на обычном хостинге/VPS и контролируемой эксплуатацией.

Ограничения:

- Инференс преимущественно на CPU.
- Предсказуемый latency и стоимость.
- Без усложнения инфраструктуры на первом этапе.
- LLM не заменяет текущий pipeline целиком, а усиливает его.

## ✅ Решение

Принять модель поставки:

1. **Runtime LLM:** `ollama` в Docker.
2. **Модель по умолчанию:** небольшой instruct-LLM (например `qwen2.5:3b-instruct`) в квантизации `Q4_K_M` или `Q5`.
3. **Интеграция:** `LlmEnricher` ходит в OpenAI-compatible endpoint (`/v1/chat/completions`) через внутренний адаптер.
4. **Режим использования:** LLM включается как fallback и/или confidence-guard, а не как единственный источник.
5. **Контракт ответа:** строго JSON по схеме (валидация обязательна, при ошибке fallback на текущую логику).

## 🧠 Почему так

- `Ollama` проще всего поднять и сопровождать на CPU.
- OpenAI-compatible API минимизирует vendor lock для приложения (можно заменить backend позже).
- Небольшая модель в `Q4_K_M/Q5` дает лучший баланс "качество/скорость/память" для задачи парсинга.
- Снижается риск регрессий: текущие правила и провайдеры остаются основой.

## ↔️ Альтернативы

### 1) LM Studio

Подходит для локальных экспериментов и ручной проверки prompt'ов, **но не выбран как production runtime**.

Минусы для прода:

- Ориентирован на desktop-сценарии.
- Хуже как серверный компонент для headless/daemon деплоя.
- Менее удобен для стандартного Docker-цикла поставки.

**Вывод:** LM Studio использовать опционально для RnD, но не как целевой runtime.

### 2) llama.cpp server

Плюсы: тонкая настройка, часто высокая CPU-эффективность.  
Минусы: больше ручной операционки, выше порог поддержки.

### 3) Внешний облачный LLM API

Плюсы: быстро стартовать без self-hosting.  
Минусы: стоимость, зависимость от провайдера, риски по приватности данных.

## 🏗️ План поставки

### Этап 0. Контракт и режим работы

1. Зафиксировать JSON-схему для `LlmEnricher` ответа:
   - `classification` (`event|meta|noise`)
   - `location_hint` (строка или `null`)
   - `confidence` (0..1)
   - `reason` (кратко, для логов)
2. Добавить hard-правила:
   - `temperature = 0`
   - ограничение `max_tokens`
   - timeout на запрос
   - mandatory JSON validation
3. Включить feature flags:
   - `RADAR_LLM_GEOCODER_ENABLED`
   - `RADAR_LLM_PROVIDER=ollama`
   - `RADAR_LLM_BASE_URL`
   - `RADAR_LLM_MODEL`

### Этап 1. Docker в локальном контуре

1. Добавить сервис `ollama` в `docker-compose` (отдельный profile, например `llm`).
2. Пробросить volume для моделей (`/root/.ollama`).
3. После старта контейнера выполнить pre-pull модели (`ollama pull ...`) через bootstrap-скрипт.
4. Подключить `worker` к `RADAR_LLM_BASE_URL=http://ollama:11434/v1`.

Критерий готовности:

- `worker` умеет отправить тестовый prompt и получить валидный JSON-ответ от локального runtime.

### Этап 2. Интеграция в pipeline (safe mode)

1. Реализовать `LlmEnricher` вместо заглушки.
2. Включить режим "не вместо, а страхующий фактор":
   - сначала текущие источники (`cache/dadata/nominatim`)
   - LLM только при неуверенном или пустом результате
3. Вести телеметрию:
   - доля вызовов LLM
   - p50/p95 latency
   - доля невалидного JSON
   - доля fallback после LLM

Критерий готовности:

- на тестовом наборе сообщений нет ухудшения базового качества, есть прирост на "сложных" кейсах.

### Этап 3. Поставка на простой хостинг (VPS)

1. Развернуть `docker compose` с профилем `llm`.
2. Ограничить ресурсы контейнера `ollama` (CPU/RAM) согласно хосту.
3. Настроить healthcheck и автоперезапуск.
4. Хранить конфиг модели в env, не хардкодить.
5. Добавить canary rollout:
   - сначала 10% сообщений через LLM
   - затем 50%
   - затем 100% при стабильных метриках.

### Этап 4. Эксплуатация

1. Логи запросов в усеченном виде (без чувствительных данных).
2. Ротация и контроль диска для model cache.
3. Ночной smoke-тест endpoint LLM.
4. Регламент rollback: переключение флага `RADAR_LLM_GEOCODER_ENABLED=0`.

## 📦 Минимальный production checklist

- [ ] LLM отвечает в заданный timeout на CPU-хосте.
- [ ] JSON-ответ всегда валидируется схемой.
- [ ] Есть fallback на текущую цепочку.
- [ ] Есть метрики качества и latency.
- [ ] Есть feature flag для мгновенного отключения.
- [ ] Документирован процесс смены модели (`Q4_K_M -> Q5` и обратно).

## ❗ Решение по LM Studio

**LM Studio не требуется для поставки.**  
Для production контура выбираем `Ollama + Docker`.  
LM Studio можно использовать только как инструмент локального тестирования prompt'ов.

# Ollama: семплинг, параметры и сравнение моделей

Практическая памятка для локальной работы с моделями (`qwen3:14b`, `qwen3:32b`) через Ollama/Open WebUI.

## Зачем этот документ

- 🧠 Быстро понять, как работает семплинг и почему ответы "плавают".
- ⚙️ Настраивать качество/скорость/стабильность осмысленно, а не случайно.
- 💾 Управлять гибридным режимом CPU+GPU под ограниченное железо.
- ☁️ Трезво сравнивать локальные 32B с облачными моделями.

## Базовая ментальная модель

LLM на каждом шаге выбирает следующий токен из распределения вероятностей.

- `temperature` меняет форму распределения (смелость выбора).
- `top_k`, `top_p`, `min_p` и похожие параметры фильтруют допустимых кандидатов.
- штрафы повторов (`repeat_*`, `presence_*`, `frequency_*`) уменьшают зацикливание.

Если сильно "зажать" все параметры сразу, текст станет сухим и однообразным. Если сильно "отпустить", ответы станут разнообразнее, но возрастет риск ошибок и галлюцинаций.

## Параметры семплинга: ликбез

### Температура и фильтры

- `temperature`  
  Чем ниже значение, тем консервативнее и повторяемее вывод.  
  Рекомендация: `0.2-0.5` для кода/аналитики, `0.6-0.9` для объяснений/черновиков.

- `top_k`  
  Оставляет только `K` наиболее вероятных токенов.  
  Малый `top_k` = стабильнее, но беднее формулировки.

- `top_p`  
  Nucleus sampling: берется минимальный набор токенов с суммой вероятностей `p`.  
  Обычно самый полезный "контроль разнообразия" вместе с `temperature`.

- `min_p`  
  Отбрасывает слишком слабые токены относительно лучшего.  
  Хороший фильтр мусорного хвоста при более творческих настройках.

- `typical_p`  
  Фильтрация по "типичности" токенов (иногда дает более естественный текст).  
  На практике чаще используют `top_p`, а `typical_p` оставляют по умолчанию.

- `tfs_z`  
  Tail free sampling: срезает дальний хвост распределения.  
  Нужен редко; обычно оставляется по умолчанию.

### Адаптивный режим Mirostat

- `mirostat` (`0`, `1`, `2`)  
  `0` выключен, `1/2` включают адаптивный контроль энтропии.

- `mirostat_tau`  
  Целевая "сложность" текста: выше = разнообразнее.

- `mirostat_eta`  
  Скорость адаптации: выше = агрессивнее корректировка.

Mirostat полезен, когда нужно более ровное поведение на длинной генерации, но чаще хватает обычной связки `temperature + top_p`.

## Антиповторы и стабильность

- `repeat_last_n`  
  Глубина окна, где ищутся повторы.

- `repeat_penalty`  
  Общий штраф за повтор токенов.

- `presence_penalty`  
  Штраф за сам факт, что токен уже встречался.

- `frequency_penalty`  
  Штраф зависит от частоты повторений.

- `penalize_newline`  
  Штраф на переводы строк. Обычно оставлять `false`, иначе ответы могут становиться "слепленными".

## Память, скорость и гибрид CPU+GPU

Это ключевые параметры для ноутбуков/desktop со средним VRAM:

- `num_gpu`  
  Сколько слоев выгружать на GPU. Главный рычаг гибридного режима.

- `num_ctx`  
  Размер контекста (очень сильно влияет на RAM/VRAM из-за KV-cache).

- `num_batch`  
  Влияет на скорость префилла и пиковое потребление памяти.

- `num_thread`  
  Количество потоков CPU.

- `main_gpu`  
  Номер основной GPU (актуально при нескольких видеокартах).

Дополнительные системные опции: `numa`, `low_vram`, `use_mmap`, `use_mlock`, `f16_kv`, `logits_all`, `vocab_only`, `embedding_only`.

## Параметры длины и воспроизводимости

- `num_predict` - лимит длины ответа.
- `seed` - фиксирует случайность (для повторяемых ответов).
- `num_keep` - сколько токенов держать при сдвиге контекста.
- `stop` - список стоп-последовательностей.

## Гибрид под `qwen3:32b` (примерный старт)

Для железа уровня `RTX 3080 mobile + 32 GB RAM`:

- `num_ctx: 2048`
- `num_batch: 32-64`
- `num_gpu: 18-24`
- `num_thread: 8-12`

Если упирается в память:
1. Снизить `num_ctx`
2. Снизить `num_batch`
3. Снизить `num_gpu`

## MCP, Tools, Skills: кто за что отвечает

Коротко: сама LLM не "подключает MCP" магически. Это делает оркестратор (клиент/харнесс).

### Роли в архитектуре

- 🧠 Модель  
  Решает, нужен ли инструмент, и формирует structured tool call (если умеет).

- 🔌 Оркестратор/харнесс  
  Передает модели schema инструментов, исполняет вызов, возвращает результат в диалог.

- 🧱 MCP-сервер  
  Реально выполняет операции (БД, файлы, API, git и т.д.).

- 📜 Rules/System prompt  
  Определяют политику: когда обязательно звать инструмент, а когда можно отвечать напрямую.

### Что нужно, чтобы модель "сама просила промежуточный вызов"

1. Поддержка function/tool-calling в модели и/или рантайме.
2. Цикл orchestration:  
   `assistant -> tool_call -> tool_result -> assistant` (до финала).
3. Guardrails в харнессе:  
   лимит шагов, таймауты, валидация JSON-аргументов, allowlist инструментов.
4. Четкий system prompt/rules (иначе модель будет нестабильна в выборе инструментов).

### Rules: нужны или нет?

- Не обязательны для "включения" tools технически.
- Но критичны для стабильного поведения.
- Рекомендуемый минимум policy:
  - если не хватает фактов -> сначала инструмент;
  - не выдумывать значения при пустом `tool_result`;
  - при конфликте источников явно указывать uncertainty.

## Qwen3:32B vs облачные модели

Короткий ответ: ощущения "очень близко" в ряде задач абсолютно нормальны. Это не иллюзия.

### Где `qwen3:32b` часто близка

- ✅ Структурированные ответы по инструкции.
- ✅ Рефакторинг и ревью средних фрагментов кода.
- ✅ Краткие архитектурные сравнения и практичные планы.
- ✅ Русскоязычные объяснения в техническом контексте.

### Где облако обычно впереди

- ☁️ Более стабильное качество на длинных многошаговых цепочках.
- ☁️ Лучше удержание сложных ограничений в длинном диалоге.
- ☁️ Сильнее инструментальный стек (tool use, экосистема интеграций).
- ☁️ Выше надежность на "краевых" и мультимодальных кейсах.

### Практический вывод

- Для локальной разработки и ежедневных задач `qwen3:32b` может давать качество "уровня облака" субъективно очень часто.
- Для критичных production-задач, где важна максимальная стабильность на длинном горизонте, облачные модели обычно надежнее.
- Лучший путь: держать локальную `qwen3` как default, а сложные/критичные кейсы при необходимости прогонять через облачную модель для second opinion.

## Bench: локалка vs облако (готовая методика)

Ниже шаблон, которым удобно сравнивать `qwen3:32b` c облаком (например, `gpt-4o`, `claude-haiku`) без субъективности.

### 1) Набор задач (минимум)

- `Task A (код)` - рефакторинг реального файла + сохранить поведение.
- `Task B (баг)` - найти и исправить дефект с edge cases.
- `Task C (архитектура)` - дать 2-3 варианта и выбрать один с trade-offs.
- `Task D (инструменты)` - задача, где нужен tool/MCP вызов.

### 2) Метрики и веса

- 🎯 Точность решения (`0-5`) - вес `35%`
- 🧠 Глубина reasoning (`0-5`) - вес `20%`
- 🔒 Надежность/не-галлюцинации (`0-5`) - вес `20%`
- ⚡ Скорость первого полезного ответа (`0-5`) - вес `15%`
- 🧰 Качество tool use (`0-5`) - вес `10%`

Итог: `Score = sum(metric * weight)`.

### 3) Шкала итоговой интерпретации

- `4.3-5.0` - практически interchangeable в твоих задачах.
- `3.6-4.2` - близко, но есть заметные ограничения.
- `2.8-3.5` - годится как рабочая локальная, но не как primary для критичных задач.
- `<2.8` - лучше использовать как вспомогательную модель.

| gpt-4o |  |  |  |  |  |  |

### 5) Что обычно показывает практика

- `qwen3:32b` часто дает очень сильный результат в коде и структурных ответах.
- Облако чаще выигрывает в длинных цепочках, нестандартных кейсах и инструментальной надежности.
- Если локалка показывает `>=4.0` на твоем наборе задач, ее вполне можно держать как основную daily-модель.

## Важно

- Значения в документе - рабочие ориентиры, а не догма.
- Для каждой модели и железа нужен короткий локальный бенч на своих задачах.
- При обновлении версии Ollama часть опций и дефолтов может меняться.

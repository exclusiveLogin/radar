# Radar — кодстайл и слои

## ESLint (`eslint.config.mjs`)

- **`@typescript-eslint/consistent-type-imports`**: `import type { X }`; не смешивать type/value без необходимости.
- **`@typescript-eslint/consistent-type-exports`**: явные type-экспорты там, где требует правило.
- **Неиспользуемое**: `_`-префикс для намеренно неиспользуемых параметров и переменных.
- Область: `packages/*/src/**/*.{ts,tsx}` через корневой `npm run lint`.

Перед финалом задачи по коду желательно мысленно прогнать соответствие этим правилам.

## Читаемость в app-слоях

Пользователь ожидает **максимально декларативный** код в основных слоях приложения:

- Короткие функции и ранние возвраты; избегать глубокой вложенности и «спагетти» из вложенных хелперов там, где достаточно линейного сценария.
- Основной поток должен читаться **сверху вниз** без необходимости в комментариях «что здесь происходит».
- Сложную логику выносить в узкие модули («глубокая реализация»), сохраняя в application/domain тонкий оркестратор.

Нейминг и импорты — в стиле соседних файлов пакета (path alias при наличии, группировка импортов как в модуле).

## Объём изменений

- Менять только то, что нужно для задачи; не делать побочный рефакторинг затронутых файлов без запроса.
- Не удалять существующие комментарии и код вне задачи только «для порядка».
- Избыток комментариев для очевидного кода не добавлять.

## Домен и инфраструктура

- **Worker parsing / domain**: правила классификации и парсинга — чистые функции по возможности.
- **API**: контроллеры тонкие; работа с БД — через репозитории и существующие паттерны TypeORM проекта.

## Связка с MCP

При примерах по **сторонним** API библиотек использовать **Context7** (не выдумывать устаревшие сигнатуры). По обширному коду репозитория допускается опора на инструменты **lean-ctx**, если MCP подключён.
